Follow conventions

Conventions are a set of rules that are commonly accepted. Sharing practices increase readability and maintenability. Conventions can be numerous, healthy ones can be found here. This is of course not a dogmatic approach, used ones may vary in practice from developer team. General view remains true though.

Good, meaningful names

Using good, meaningful names is important to improve readability and understanding of current context. For exemple, if a user already received a notification, you should prefer userHasBeenNotified to notified. Descriptive approach should be favored. Meaningful names should be applied to everything: functions, class names, variables, ...

KISS principle

Keep It Simple, Stupid! (I'm sorry). This principle states that code should be kept as simple as possible. Complex code can be error prone, time consumming, and difficult to read. Code should be written in the most simple and logic way (which does not mean the shortest), without any unnecessary complexity.

Scalability

Scalability is the possiblity to modify, add new things without having to change existing ones. Same previous exemple, userHasBeenNotified is a scalable name. If you want to add an admin category, you would just replace 'user' with 'admin'. Scalability is applied as well for variable names as for code logic and can save you valuable time.

DRY principle

DRY principle (Don't Repeat Yourself) is a key aspect (if not the most important one) of clean code. It states that code duplication should be avoided. Usually, if we repeat a portion of code, we should externalise it in a function. Doing this will increase readability and maintenability: if this code has to be changed, it will be in one place. DRY principle is applied to everything, HTML and CSS included. Beeing 'DRY' often increases the abstraction level, so a tradeoff between abstraction and readability has to be done. You don't want an ultra-optimised but hard to read code.
